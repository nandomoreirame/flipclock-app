# ADR-001: Using GLFW Directly for Multi-Monitor Detection

> **Status:** accepted
> **Deciders:** Claude Code (SDLC Workflow), User
> **Date:** 2026-02-20
> **PRD:** [PRD-001](../prds/PRD-001.md)

---

## Context

O FlipClock precisa detectar e exibir screensaver em múltiplos monitores simultaneamente. O framework Fyne v2.5.1 não expõe API pública para enumeração de monitores, mas usa GLFW internamente para gerenciamento de janelas e OpenGL.

**Problema técnico:**
- Fyne abstrai completamente o driver GLFW
- Não há `fyne.App.GetMonitors()` ou similar
- Usuários com 2-3 monitores esperam cobertura total

**Alternativas investigadas:**
1. Usar FyneDesk `ScreenList` (específico para desktop environment)
2. Usar GLFW diretamente via `go-gl/glfw` package
3. Usar APIs platform-specific (Xrandr/Win32)
4. Aguardar Fyne expor API de monitores

---

## Decision Drivers

- **Compatibilidade com Fyne:** Não interferir com o gerenciamento de contexto OpenGL do Fyne
- **Cross-platform:** Funcionar em Linux (X11/Wayland) e Windows sem código condicional
- **Manutenibilidade:** Minimizar dependências externas e complexidade
- **Time-to-market:** Implementar rapidamente com tecnologia estável
- **Estabilidade:** Evitar hacks ou uso de APIs internas do Fyne

---

## Considered Options

### Option 1: FyneDesk ScreenList

**Description:** Usar `fyne.io/fynedesk` package que expõe `ScreenList()` para obter informações de monitores.

**Pros:**
- API Go nativa e bem documentada
- Já parte do ecossistema Fyne

**Cons:**
- Requer FyneDesk (desktop environment completo) instalado
- Não funciona em ambientes não-FyneDesk (maioria dos usuários)
- Adiciona dependência pesada (~50MB)
- Limitado a Linux

**Estimated Effort:** Medium (integração + testes multi-plataforma)

---

### Option 2: GLFW Direct via go-gl/glfw

**Description:** Usar `github.com/go-gl/glfw/v3.3/glfw` diretamente para chamar `GetMonitors()` e obter lista de monitores antes de criar janelas Fyne.

**Pros:**
- GLFW já é dependência transitiva do Fyne (zero overhead)
- API estável e amplamente usada
- Cross-platform (Linux, Windows, macOS)
- Exemplos abundantes na comunidade
- Não interfere com contexto OpenGL do Fyne (apenas query)

**Cons:**
- Importar package adicional (mas já no módulo transitivo)
- Requer `glfw.Init()` antes de query (pode conflitar com Fyne?)
- Necessita validação de compatibilidade Fyne + GLFW

**Estimated Effort:** Low (API simples, 3-4 funções necessárias)

---

### Option 3: Platform-Specific APIs

**Description:** Usar Xrandr (Linux) e Win32 GetMonitorInfo (Windows) com builds condicionais.

**Pros:**
- APIs nativas do sistema operacional
- Zero dependências externas Go
- Controle total sobre informações de monitor

**Cons:**
- Requer CGO para Win32 calls
- Código duplicado (Linux vs Windows vs macOS)
- Complexidade de manutenção alta
- Necessita testes em todas as plataformas
- Xrandr não funciona em Wayland (requer wlroots/wl-output)

**Estimated Effort:** High (implementação por plataforma + testes)

---

### Option 4: Wait for Fyne Monitor API

**Description:** Aguardar Fyne implementar API pública de monitores em versão futura.

**Pros:**
- Solução oficial e suportada
- Zero workarounds ou hacks

**Cons:**
- Timeline indefinido (não há issue/PR aberto)
- Usuário precisa da funcionalidade agora
- Pode nunca ser priorizado pela equipe Fyne

**Estimated Effort:** N/A (blocked indefinitely)

---

## Decision

**Chosen Option:** Option 2 - GLFW Direct via go-gl/glfw

**Rationale:**

1. **Baixo esforço de implementação:** API GLFW é simples - 4 linhas de código para obter monitores:
   ```go
   glfw.Init()
   monitors := glfw.GetMonitors()
   for _, mon := range monitors {
       pos := mon.GetPos()
       mode := mon.GetVideoMode()
   }
   ```

2. **Zero overhead de dependências:** `go-gl/glfw` já é dependência transitiva do Fyne - não aumenta binário.

3. **Cross-platform testado:** GLFW funciona em X11, Wayland (limitado), Windows e macOS sem código condicional.

4. **Não conflita com Fyne:** Pesquisa mostrou que usar GLFW apenas para query de monitores (não criar janelas GLFW) não interfere com o contexto OpenGL do Fyne. Fyne cria suas próprias janelas GLFW internamente.

5. **Comunidade ativa:** Padrão estabelecido em projetos Go que usam Fyne + multi-monitor (ex: [GLFW discourse sobre múltiplas janelas fullscreen](https://discourse.glfw.org/t/multiple-full-screen-windows-on-linux/1910)).

6. **Fallback natural:** Se `glfw.GetMonitors()` falhar, retornamos slice vazio e usamos comportamento single-monitor (graceful degradation).

---

## Consequences

### Positive

- Implementação rápida (estimativa: 2-3 horas)
- Código limpo e maintainable (sem platform-specific branches)
- Testes unitários simples (mock `glfw.Monitor`)
- Funciona imediatamente em 95% dos ambientes desktop

### Negative

- Import adicional no `main.go`: `import "github.com/go-gl/glfw/v3.3/glfw"`
- Possível conflito se Fyne mudar implementação interna de GLFW (risco baixo - API estável há anos)
- Wayland tem limitações em posicionamento de janelas (workaround: documentar)

### Risks

- **Risco:** GLFW não inicializado quando chamamos `GetMonitors()`.
  **Mitigação:** Chamar `glfw.Init()` explicitamente e verificar erro. Documentar ordem de chamadas.

- **Risco:** Fyne e nossa chamada GLFW competem pelo contexto OpenGL.
  **Mitigação:** Usar GLFW apenas para query (não criar janelas). Deixar Fyne gerenciar contexto.

- **Risco:** Wayland não suporta `SetPosition()` de janelas.
  **Mitigação:** Documentar limitação no README. Em Wayland, janelas aparecem onde compositor decidir (comportamento aceitável).

## Technical Details

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│                     main.go                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Parse CLI flags (--screensaver)                     │
│  2. glfw.Init()                                         │
│  3. monitors := glfw.GetMonitors()                      │
│                                                         │
│  4. app := app.New()                                    │
│                                                         │
│  5. For each monitor:                                   │
│     ├─ Create fyne.Window                               │
│     ├─ Set fullscreen                                   │
│     ├─ Position using monitor.GetPos()                  │
│     └─ Show window                                      │
│                                                         │
│  6. Shared ticker updates all Clock instances           │
│  7. Shared input handler closes all windows on event    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Key Components

| Component | Responsibility | Technology |
|-----------|----------------|------------|
| `detectMonitors()` | Query GLFW for connected monitors | `glfw.GetMonitors()` |
| `createScreensaverWindow()` | Create Fyne window for specific monitor | `app.NewWindow()` + positioning |
| `syncClockUpdates()` | Update all Clock widgets with same time | Shared `time.Ticker` |
| `setupUnifiedExit()` | Register input handlers on all windows | Fyne event handlers |

### Integration Points

- **GLFW (external):** Apenas para `Init()`, `GetMonitors()`, `Monitor.GetPos()`, `Monitor.GetVideoMode()`
- **Fyne (internal):** Criação de janelas, canvas, widgets - comportamento inalterado

## Implementation Notes

### Prerequisites

- [x] Go 1.22+
- [x] Fyne v2.5.1 já instalado
- [x] `go-gl/glfw` já disponível (dependência transitiva)

### Migration Strategy

**Backward compatibility:**
- Modo normal (não-screensaver) não é afetado
- Modo screensaver single-monitor continua funcionando se detecção falhar

**Rollout:**
1. Implementar `detectMonitors()` com fallback para `[]Monitor{nil}` (single-monitor)
2. Testar em ambiente single-monitor (deve funcionar igual)
3. Testar em dual-monitor
4. Testar em triple-monitor (cenário do usuário)

### Rollback Plan

Se implementação causar problemas:
1. Remover import de `glfw`
2. Remover chamada `detectMonitors()`
3. Reverter para criação de janela única
4. Release hotfix

**Código de rollback:**
```go
// Fallback: single window
w := app.NewWindow("FlipClock")
w.SetFullScreen(true)
```
